# Директория .specify

Эта директория содержит документацию проекта, спецификации и инструменты для рабочего процесса спецификационного разработки (SpecKit).

## Структура директории

```
.specify/
├── memory/          # Память проекта и конституция
├── specs/           # Спецификации функций
├── templates/       # Шаблоны документации
├── scripts/         # Вспомогательные скрипты
└── docs/            # Документы анализа и резюме
```

## Поддиректории

### memory/

Содержит долгосрочную память проекта и конституционные документы:

- **constitution.md** - Основные принципы проекта, архитектурные стандарты, технологические руководства и дорожная карта разработки. Это авторитетный источник проектных решений.

### specs/

Спецификации функций, организованные по номеру и имени (NNN-feature-name):

```
specs/
└── 001-nextjs-foundation-setup/
    ├── spec.md          # Спецификация функции
    ├── plan.md          # Технический план реализации
    ├── tasks.md         # Разбивка задач
    ├── data-model.md    # Модель данных (если применимо)
    ├── research.md      # Исследования и решения
    ├── quickstart.md    # Сценарии тестирования
    ├── checklists/      # Контрольные списки валидации
    └── contracts/       # API контракты и схемы
```

Каждая функция получает свою нумерованную директорию (001, 002, 003 и т.д.) с описательным именем.

### templates/

Шаблоны документации для единообразной структуры:

- **spec-template.md** - Шаблон для спецификаций функций
- **plan-template.md** - Шаблон для технических планов
- **tasks-template.md** - Шаблон для разбивки задач
- **checklist-template.md** - Шаблон для контрольных списков валидации
- **agent-file-template.md** - Шаблон для инструкций агентов

### scripts/

Вспомогательные скрипты для автоматизации рабочего процесса:

- **bash/check-prerequisites.sh** - Валидация окружения функции
- **bash/create-new-feature.sh** - Создание структуры новой функции
- **bash/setup-plan.sh** - Инициализация документов планирования
- **bash/update-agent-context.sh** - Обновление инструкций агентов
- **bash/common.sh** - Общие функции и утилиты

### docs/

#### docs/analysis/

Документы архитектурного и кодового анализа:

- Документы архитектурного анализа (ARCHITECTURAL-ANALYSIS*.md)
- Результаты проверки кода (REVIEW-FINDINGS*.md)
- Оценка технического долга
- Анализ паттернов

#### docs/summaries/

Резюме проектов и предложения:

- Резюме интеграции лучших практик
- Резюме улучшения спецификаций
- Резюме завершения задач
- Предложения по обновлению конституции
- Резюме планирования

## Использование

### Для разработчиков

1. **Начало новой функции**: Используйте команду `/speckit.specify` или запустите `create-new-feature.sh`
2. **Планирование реализации**: Используйте `/speckit.plan` для генерации технического плана
3. **Разбивка задач**: Используйте `/speckit.tasks` для создания списка задач
4. **Реализация**: Используйте `/speckit.implement` для выполнения задач
5. **Анализ**: Используйте `/speckit.analyze` для проверки согласованности

### Для агентов

Агенты, работающие в режимах **tasks** и **analytics**, имеют полный доступ к директории `.specify/`:

- **Чтение**: Вся документация, спецификации и шаблоны
- **Запись**: Поддиректории specs/ и docs/ для обновлений
- **Выполнение**: scripts/ для автоматизации

### Для проверяющих

Перед проверкой изменений кода:

1. Проверьте `.specify/memory/constitution.md` для принципов проекта
2. Прочитайте соответствующую спецификацию функции в `.specify/specs/NNN-feature-name/spec.md`
3. Просмотрите технический план в `.specify/specs/NNN-feature-name/plan.md`
4. Проверьте контрольный список задач в `.specify/specs/NNN-feature-name/tasks.md`

## Стандарты документации

Вся документация в этой директории следует следующим стандартам:

1. **Двуязычная**: Английский - основной, русские переводы предоставлены где применимо
2. **Формат Markdown**: Все документы используют Markdown (.md)
3. **Структурированная**: Следуйте шаблонам для согласованности
4. **Версионирована**: Отслеживайте изменения через git
5. **Перекрестные ссылки**: Связывайте связанные документы

## Команды SpecKit

Рабочий процесс SpecKit предоставляет следующие команды:

- `/speckit.specify` - Создать/обновить спецификацию функции
- `/speckit.clarify` - Уточнить неоднозначные требования
- `/speckit.plan` - Сгенерировать технический план реализации
- `/speckit.tasks` - Разбить на выполнимые задачи
- `/speckit.analyze` - Валидировать согласованность артефактов
- `/speckit.implement` - Выполнить реализацию
- `/speckit.constitution` - Обновить конституцию проекта

## Соглашения об именовании файлов

### Директории функций

Формат: `NNN-краткое-описательное-имя`

Примеры:
- `001-nextjs-foundation-setup`
- `002-user-authentication`
- `003-analytics-dashboard`

### Документы анализа

Формат: `ОПИСАТЕЛЬНОЕ-ИМЯ[-ЯЗЫК].md`

Примеры:
- `ARCHITECTURAL-ANALYSIS.md` (Английский)
- `ARCHITECTURAL-ANALYSIS-RU.md` (Русский)
- `REVIEW-FINDINGS.md`

### Документы резюме

Формат: `ТЕМА-SUMMARY[-ЯЗЫК].md`

Примеры:
- `PLANNING-SUMMARY.md`
- `TASK-COMPLETION-SUMMARY.md`
- `BEST-PRACTICES-INTEGRATION-SUMMARY-RU.md`

## Обслуживание

### Добавление новых функций

Запустите скрипт создания:
```bash
.specify/scripts/bash/create-new-feature.sh "Описание функции"
```

Это:
1. Определит следующий доступный номер
2. Создаст структуру директории функции
3. Инициализирует spec.md из шаблона
4. Создаст git ветку (если в git репозитории)

### Обновление конституции

Используйте агента конституции:
```
/speckit.constitution Обновить аутентификацию для использования новой библиотеки
```

### Организация документации

- Храните документы анализа в `docs/analysis/`
- Храните резюме в `docs/summaries/`
- Архивируйте старые документы вместо удаления
- Обновляйте этот README при добавлении новых типов документов

## Интеграция с GitHub

Директория `.github/agents/` содержит инструкции агентов, ссылающиеся на `.specify/`:

- Агенты читают конституцию из `.specify/memory/constitution.md`
- Агенты читают/пишут спецификации в `.specify/specs/`
- Агенты используют шаблоны из `.specify/templates/`
- Агенты выполняют скрипты из `.specify/scripts/`

## Лучшие практики

1. **Всегда работайте через рабочий процесс SpecKit**: Не редактируйте спецификации вручную без соответствующих команд
2. **Поддерживайте конституцию актуальной**: Документируйте важные архитектурные решения
3. **Архивируйте, не удаляйте**: Перемещайте старые документы в соответствующие поддиректории
4. **Используйте шаблоны**: Поддерживайте согласованность между функциями
5. **Перекрестные ссылки**: Связывайте связанные документы
6. **Контроль версий**: Фиксируйте изменения регулярно с описательными сообщениями
